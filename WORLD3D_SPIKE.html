1. What this is

A spike that answers one product question: can you actually work in a 3D "world" of your terminals? It takes the real, live termtastic panes and arranges them in a ring (a mini "rotunda", inspired by the 3D app switcher). The pane rotated to the front is a genuine, fully-interactive terminal — real shell, real line editing, real mouse selection — while the others stay live but passive around it. You can rotate the ring and zoom the front pane.

It is deliberately a spike, not a feature: one window's currently-mounted panes, a plain ring, a temporary topbar button. But it exercises the real feature's core and retired every hard technical unknown. It is not merged — it lives only on the world3d-spike branch, gated behind a temporary cube button in the topbar (next to the globe / 3D switcher button).

The headline result: a real xterm.js terminal on a CSS3D 3D plane can be focused, typed into, and selected accurately, composited with the rest of the 3D scene — as long as the interactive pane is kept flat-facing the camera at a true 1:1 pixel mapping. That single invariant drives most of the design.

3. How it works (architecture)

  • Open: enumerate the real mounted panes (terminals.values whose container.isConnected). For each, reparent its real xterm container (TerminalEntry.container, a .terminal div) out of the 2D layout and onto a CSS3DObject plane, remembering its original parent/sibling.
  • Layout: planes sit on a horizontal ring (RING_R). Each frame, pane i is placed at angle baseAngle[i] + ringAngle, facing outward. The front pane (angle ≈ 0) faces the camera.
  • Camera: a single PerspectiveCamera, fixed at the CSS3D "perspective distance" (perspDistance()) so a plane at the ring front, unscaled, renders at exactly 1:1 with its native pixels. Both the CSS3D renderer and (conceptually) the scene share it.
  • Interactivity: only the settled front pane gets pointer-events:auto + focus; it is the real terminal, so backspace / word-motion / history / selection all work. Others get pointer-events:none and a dim overlay (still live — they're the actual terminals, just passive).
  • Rotate: ◀ ▶ buttons or Alt+←/→ change which pane is front; the ring eases; the new front takes focus.
  • Zoom: Alt+↑/↓ zoom the front pane (see the zoom lesson below).
  • Close: restore every terminal's native font, then move each container back to its exact original DOM position. The app's 2D layout is untouched afterward.

4. Key mechanisms & hard-won lessons

Each of these was a real bug or dead-end. Read them before changing the corresponding area — the "obvious" simplification usually reintroduces the bug.

Why CSS3D (not a WebGL texture)

Problem
The 3D switcher renders panes as snapshot textures — read-only. You can't type into a texture.
Insight
three.js's CSS3DRenderer renders real DOM elements as CSS-transformed 3D planes, sharing the camera. So a real xterm keeps full input/cursor/selection.
Solution / caveat
Reparent the real xterm onto a CSS3DObject. Compositing rule: the whole CSS3D layer is one browser compositing plane — WebGL geometry cannot be drawn in front of a CSS3D element (they're stacked layers). CSS3D elements do occlude each other. That's why this spike is pure CSS3D (no WebGL layer).

The 1:1 selection invariant (the load-bearing constraint)

Problem
xterm maps a mouse position to a cell by dividing the pointer offset by the layout cell size. Any CSS scale or rotation/tilt breaks this: a scaled plane's on-screen size ≠ its layout size, so selection drifts (measured: at 3× visual zoom, a drag at y=271 selected at y=374, and the error grows down the pane).
Insight
Selection is only correct when 1 layout px = 1 screen px for the interactive pane — i.e. flat-facing the camera, unscaled, at the perspective distance.
Solution
The front pane is billboarded flat and its CSS3DObject scale is eased to 1.0. Tilt/scale is only ever applied to non-interactive side panes. If you ever need selection on a tilted pane, you must bypass xterm's mouse math entirely (raycaster → UV → cell → drive selection) — see Next Steps.

Reparenting real terminals

How
Move entry.container into the plane wrapper; capture origParent + origNext and the prior inline CSS (stashed on a data-spike-prevcss attribute) to restore exactly on close.
Rule
An xterm's element can live in only one DOM parent, so this is "the actual pane" (not a copy). Pin the container to its current px size so it doesn't refit — refitting resizes the real PTY (server churn) and reintroduces the gray-margin bug below.

The gray "OOB unused-space" bars (only on real terminals)

Problem
Real terminals showed gray bars; demo (emulated) didn't. A terminal grid is quantised to whole character cells, so it can't fill an arbitrary-pixel box — leaving a sub-cell margin. Real, server-sized PTYs shade that margin with a gray .oob-overlay ("Reformat to reclaim"); demo terminals refit to fill, so no margin.
Solution
Size the plane to the actual grid (.xterm-screen rect), pin the container to its native size, offset it so the grid sits at the plane's top-left, clip the remainder (overflow:hidden), and hide oobOverlayRight/Bottom. No unused space → nothing to shade. See sizePlaneToGrid().

Zoom: the whole saga (smooth AND exact selection)

Attempt 1 — camera dolly
Smooth, but visually scaled the front pane → broke selection (see 1:1 invariant).
Attempt 2 — font-size zoom
Front stays 1:1 (selection exact), but every Alt+↑/↓ reflowed the terminal → janky.
Insight (from the repo owner)
Decouple the two: scale smoothly during the gesture, commit a crisp font on settle. And snap zoom steps to whole font sizes so the commit lands exactly (no quantization pop).
Solution (current)
  • Alt+↑/↓ step a target zoom by one whole font size (zoom = fs / baseFont).
  • During the gesture the front pane eases its CSS3DObject scale toward target/committed — pure GPU transform, no reflow, fluid.
  • ZOOM_COMMIT_MS (200ms) after the last press, commitZoom() re-bakes the font to the target (crisp, 1:1) and divides the visual scale by the same grid-growth ratio in the same frame → on-screen size unchanged across the swap; the loop eases the residual scale to 1.0.
  • Because the container is scaled with the font, the column/row count stays ~constant (minimal PTY churn), and because the target is a font-achievable zoom, committed == target exactly → seamless landing.
Net: smooth zoom + pixel-exact selection at rest. Selection is only "off" during the ~200ms active gesture, which is fine (you're zooming, not selecting).

Side-pane normalization & interaction model

Normalize
Non-front panes ease toward a common height (COMMON_H), shrink-only (clamped ≤ 1.0) to avoid upscaling blur. The interactive front pane is never scaled (1:1 invariant).
Keys
All navigation uses Alt+arrows so plain arrow keys still reach the focused shell (the classic terminal-in-3D tension: every keystroke is meaningful, so you can't use bare arrows for camera control).

7. How to run & verify

Run in the browser (demo, no backend):

cd <this worktree>
./gradlew :web:jsBrowserDevelopmentRun --continuous     # serves http://localhost:8080
# open http://localhost:8080/?demo  → click the cube icon in the topbar

Run against real terminals: open the app normally (not ?demo) with a few panes; click the cube. Only then do you get true interactive shells (demo panes are a scripted sim). Real terminals are also where the gray-bar and font/selection edge cases actually bite — always sanity-check there.

Verification harness (important — this is how every fix above was confirmed). Headless Chrome + demo mode + CDP:
  • Launch: Google Chrome --headless=new --remote-debugging-port=9222 --enable-unsafe-swiftshader --use-gl=angle --use-angle=swiftshader (swiftshader is required — WebGL/CSS3D).
  • Node 22+ has a global WebSocket; a small .mjs speaks CDP directly (create target via PUT /json/new, then Runtime.evaluate / Input.dispatchKeyEvent / Input.dispatchMouseEvent / Page.captureScreenshot).
  • Key CDP gotchas: modifiers are a bitmask (Alt=1), not altKey; printable chars use the text field; strip #webpack-dev-server-client-overlay before screenshots.
  • Selection test that caught the zoom bug: drag across a known line on the front pane, then read the .xterm-selection child rects and compare their y to the drag y. If they diverge, selection is broken. (The front pane is the wrapper with pointer-events:auto — not the last .xterm on the page.)
See docs/web-demo-headless-verify notes; the driver scripts used during development lived in a scratchpad (ephemeral) — re-create as needed.

8. Known limitations

  • Selection during an active zoom gesture is off for ~200ms (the pane is visually scaled until commit). By design. If precise selection while visually scaled is ever required, only the raycaster approach solves it.
  • Only one window's currently-mounted panes are shown. Unmounted tabs (which the switcher previews via mirror sockets) are ignored.
  • Side-pane normalization is invisible with < 3 panes (the demo often has 2; the second sits directly behind the front). Test with 3+.
  • Reparent may resize real PTYs if a refit fires (we avoid it by pinning sizes, but font-zoom does a controlled refit).
  • Differently-proportioned panes make the ring slightly irregular; normalization is height-only.
  • No click-a-side-pane-to-front yet (side panes are pointer-events:none); rotation is via keys/buttons only.

9. Next steps / ideas

  • Pull panes from all tabs, not just the mounted ones, so the ring is fuller (reuse the switcher's mirror-source pattern for unmounted tabs).
  • Click a side pane to bring it to front (needs a pick path that doesn't steal terminal input — e.g. a transparent hit target, or the raycaster).
  • Raycaster selection for tilted panes: add an invisible WebGL pick-plane matching each pane's transform, raycaster.intersectObject().uv → (col,row), and drive xterm selection — the only way to select on a non-1:1 pane. Note xterm's public selection API is thin (single-row select, whole-line selectLines); arbitrary multi-row ranges need its internal SelectionService.
  • Persist pane world positions (LocalStore JSON, same pattern as the toolkit's pane geometry) → promote from "ring" to a freely-arranged spatial workspace.
  • The "cockpit wall" model: park the panes you monitor on a curved wall in view (all live), scatter the rest across the world, and surface off-screen activity with beacons + a mini-map. This reconciles "watch many at once" with "store panes in different places".
  • Promote to a real feature: a settings toggle + persisted layout + a proper entry point (replace the temporary cube button), and a "focus animates to face you at 1:1" transition so previews can tilt while the focused pane stays selectable.

11. Gotchas for the next agent (Kotlin/JS specifics)

  • Kotlin block comments nest. A literal /* inside a KDoc (e.g. writing a path like examples/jsm/*) opens a nested comment → "Unclosed comment". Avoid /* in doc text.
  • Control chars in strings (ESC 0x1B, DEL 0x7F) must be real bytes or unicode escapes (\u001b, \u007f) — a literal ANSI code like [1;36m written without the ESC just prints the bracket codes as text.
  • CSS3D addon import: @JsModule("three/examples/jsm/renderers/CSS3DRenderer.js") resolves under the repo's webpack (three 0.170).
  • Typed DOM gaps: CSSStyleDeclaration lacks pointerEvents in the binding — use style.setProperty("pointer-events", …). Vector props (position/rotation/scale/up) are dynamic — use .set(x,y,z) and read .x as Double.
  • json(...) is kotlin.js.json (needs the import); used for three.js/xterm option objects.
  • .terminal selector over-matches in tests (nested xterm elements) — count real panes via .xterm or the reparented wrappers, not .terminal.
  • Camera fov/up are reset per open in the core (Overview3D.openOverview3d) so a style that animates them can't leak — the spike relies on up=(0,1,0) and the fixed distance.
