3D World Grid Sizing & Reformat

2026-07-06 branch: world3d-spike session: fix-3d-world-reformat-blanking
Current state

Two sessions, nine rounds — a swallowed TypeError aborting every resize (R4), three latent bugs in the never-before-seen OOB overlay it awakened (R5–R6), a fleet of phantom headless clients plus a reconnect-ForceResize eviction storm squeezing every shared PTY (R7), a fit-vs-grid feedback ratchet in the 3D world that the R4 fix ironically completed (R8), and a second session (R9) that made reformat real (the jiggle), made grid resizes fluid, gated the last two ratchet paths, and finally made the pane box tell the truth about its chrome. overlay bars confirmed gone in-app R8 verified R9 verified in-app: "seems to work, after a moment it settles correctly"

The fixes now live in two places: this spike branch (dev bundle for scripts/run-electron-3d.sh) and a resize-fixes worktree off main (…/termtastic/resize-fixes, commits 57bbbe0, b5837cb, 42c6c38) that the user now daily-drives as their production server. A related focus fix landed in darkness-toolkit (d1df34e): maximizing a pane now makes it active + focused.

Goal

In the 3D world, the grid keys (, . width, < > height) should immediately add or remove columns/rows so the visible pane grows or shrinks, with the reformat applied automatically — not hold the pane at one display size while changing how much fits. Reformat (r) should work without destroying content, and nothing in 3D should damage the 2D view of the same session.

Root causes found
  • Two competing size authorities. The pane's pixel box (uniform-screens look) and the PTY's actual grid were reconciled through FitAddon fits and .xterm-screen rect measurement ("hug"). xterm updates its DOM asynchronously, so these reads were stale — panes failed to resize, jumped size on reformat, opened blank. This was the bug factory.
  • Server: forceClientSize evicts everyone. A ForceResize clears every client's size vote, pins the PTY to the sender's grid, and broadcasts Size to all attached clients — so a 3D mirror reformat collapsed the real 2D terminal (TerminalSessionManager.kt:473).
  • Mirrors ignored Size frames. The preview socket dropped all text frames, so a mirror's grid drifted from the PTY and repaint bytes rendered against the wrong grid (World3DSpikeRing.kt).
  • SIGWINCH repaint wipes a mirror's view. A mirror shows only a replayed scrollback tail (fresh xterm, 400-line scrollback, 64 KB ring replay); the program's repaint on resize clears it. Only alt-screen TUIs were guarded.
  • World open was not PTY-neutral. Opening the world silently fitted every mounted pane to the box and sent resize votes — perturbing live sessions on every entry.
  • Working-border SVG built once at pane-birth size, never resized — its travelling accent-coloured dot drifted off the real pane edge (the pink dot).
  • Teardown never restored 2D geometry — a 3D-forced PTY size survived back into 2D.

ROUND 3

Rethink: the PTY-truth sizing model

shipped, unverified

One size authority: the PTY grid. A pane is a truthful window onto it — pane pixels are derived arithmetically (cols × cellWidth, rows × cellHeight), never measured from the DOM. The grid only changes on explicit command, and every change is one synchronous, atomic step: terminal + pane plane + border SVG + PTY together.

  • , . = ±10 columns; < > = ±4 rows (user-approved steps)
  • r = set grid to fill the standard screen box, computed arithmetically
  • Server Size broadcasts flow through the same function (follow, don't reassert)
  • World open is PTY-neutral — no fits, no votes, panes appear at their true size (user approved dropping the uniform-at-open look, which was a lie anyway)
  • No per-pane grid memory needed — the PTY itself remembers
  • Working-border SVG resizes with the pane (pink dot fix)
  • Deleted: hugGrid, applyGridScale, factor maps, GRID_SCALES_BOX, the re-hug timer, all FitAddon use in the 3D grid path
Outcome: compiled clean; all new symbols verified in the served bundle; old symbols verified gone. In-app test: grid keys still changed only the internal terminal size — the pane box never moved. The model was right; something was silently aborting its presentation step.
ROUND 4

The actual culprit: a hidden TypeError inside every resize

root-caused, fixed, verified headless

Reproduced the symptom in headless Chrome against the demo client (?demo + CDP): pressing . grew the xterm screen (970 → 1223 px) while the pane wrapper stayed pixel-identical, with zero width/height style writes. Debugger.setPauseOnExceptions then caught what runCatching had been swallowing:

  • TypeError: tmp0_safe_receiver.asDynamic is not a function in updateOobOverlay (TerminalHelpers.kt:186): entry.term.asDynamic().element?.asDynamic()?.querySelector(…) — a Kotlin/JS trap: ?.asDynamic() on an already-dynamic receiver compiles to a real .asDynamic() method call on the DOM element at runtime, which throws whenever the terminal has a live element (i.e. always). Confirmed the compiled bundle contained exactly one such runtime call site — this one. The line is old committed code, matching "reformat has never really worked in 3D".
  • xterm fires onResize listeners synchronously inside term.resize(); the 2D layout wires onResize → updateOobOverlay. So every setPaneGrid ran term.resize() (grid changed ✓), the listener threw, the exception propagated out of resize(), and runCatching silently skipped presentPaneToGrid — internal size changed, pane box never did. The same throw also broke applyServerSize after its own resize (it calls updateOobOverlay before spikeOnServerSize), so Size-broadcast following died too.

Fixes: (1) TerminalHelpers.kt — cast to HTMLElement first, no ?.asDynamic() on a dynamic receiver; (2) World3DSpikeReformat.kt — setPaneGrid isolates term.resize() in its own runCatching so no listener anywhere can ever abort the presentation again.

Outcome: verified headless in demo mode — .. grew the pane 751→852 px, , shrank it, > grew rows, r snapped to the box with content intact. Side effect: the OOB "unused space" hatch overlays render for the first time ever (the function used to die before drawing them) — surfacing a chain of latent bugs in that never-exercised feature, fixed across Rounds 5–6 — and, less obviously, completing the Round-8 feedback loop.
ROUND 5

The awakened OOB overlays: slivers + never-clearing hatch

fixed, in served bundle
  • Hatch didn't clear on 2D Reformat. updateOobOverlay measured the used area from .xterm-screen's getBoundingClientRect() — but xterm repaints its DOM asynchronously, so when the overlay recomputed (synchronously, mid-resize) it saw the stale pre-reformat size and nothing ever re-measured. Same stale-DOM class of bug as the whole 3D saga. Now sized arithmetically from the live grid (cols × cellWidth), correct at the instant of the resize.
  • Slivers on healthy panes. The show-threshold was 4 px, but a perfectly fitted terminal always leaves a sub-cell remainder (up to one cell) — now the threshold is one full cell per axis.
  • Why the hatch appears at all right now: it marks PTYs genuinely pinned smaller than their 2D boxes (leftover ForceResizes from today's 3D work — the condition existed before; it was just invisible because the overlay code always crashed). The installed production app's bundle (server.jar, built 07-05) carries the same crash, so it can never show them — they appear only through the dev-bundle client. One Reflow per affected pane re-fits and re-pins the PTY; with the stale-rect fix the hatch now actually disappears when it does.
Outcome: improved but not enough — bars persisted in-app (bottom-only in the end). Two deeper overlay bugs remained; see Round 6.
ROUND 6

Overlay endgame: judge by what Reformat would actually do

fixed — bars confirmed gone in-app
  • Padding counted as "unused". The overlay compared the used grid against the pane container's full rect — but the container includes structural chrome the terminal can never occupy (the xterm element's padding, the scrollbar gutter, the sub-cell remainder: ~15–25 px). Every perfectly fitted pane showed permanent bars, and Reformat "did nothing" because there was nothing to reclaim. Fix: gate on the fit addon's proposeDimensions() — show only when a reformat would add ≥1 col/row.
  • Then bottom-only bars remained: the gate used the fit addon's raw proposal, but the actual Reformat applies safeFit, which clamps rows against the real viewport height (xterm one-row-overshoot guard). The overlay demanded a row Reformat correctly refuses to add. Fix: extract the clamped computation as proposeSafeDimensions() and use it for both the fit and the overlay gate — one source of truth, so the overlay lights up iff Reformat would change the grid.
Outcome: user confirmed "the bars are gone" on the daily-driver build. Overlays now appear only when another client genuinely holds the session smaller — and Reformat clears them by construction.
ROUND 7

Phantom clients + the reconnect-ForceResize eviction storm

root-caused & fixed

Reformat in 2D "did nothing" against the live server even after Round 6. lsof told the story: a headless Chrome left running by a concurrent Claude session held three hidden pages logged into the production server (756×469 window, live xterms, 21 established connections) — surviving even a server restart, since clients auto-reconnect every 500 ms.

  • Server sizes each PTY as min() across attached clients' votes (applyEffectiveSize) — tmux-style, by design. Small background clients legitimately shrink shared sessions.
  • The real bug: every pane socket sent a ForceResize on every (re)connect (connectPane.onopen → forceReassert) — the eviction kind that wipes all other clients' votes and pins the PTY. Last connector wins; the phantom's reconnect churn bulldozed every session seconds after any user Reformat. Confirmed present in the production (main) bundle too.
  • Fix: reconnect now fits locally and casts a soft vote (min() semantics preserved; cold-startup still works — min over one vote is that vote). The manual Reformat button keeps force semantics: an explicit user action. Plus the server floors forced sizes at 20×5 so a degenerate force can never collapse all clients.
  • Known trade-offs, watched during daily-driving: a session stays small while a genuinely smaller client is attached (honest min(), not a bug); a dead client's vote could linger if the server misses the disconnect — mitigated by the server's 20 s WebSocket pings (~40 s teardown).
Outcome: phantom killed (kill 59803); soft-vote + floor fixes ported to the resize-fixes worktree the user now daily-drives.
ROUND 8

The 3D shrink ratchet — completed by the Round-4 fix

fixed, awaiting in-app verification

New symptom after everything above: entering the 3D world made panes shrink step by step to tiny — in every attached client at once. A feedback loop between the two sizing philosophies:

  • In 3D, a pane's container is derived from the grid (presentPaneToGrid: exactly cols × cellW).
  • The 2D ResizeObserver still watched that container: reparent → autoReflow fit → the fit addon subtracts its padding/scrollbar allowance → proposes a smaller grid → votes it to the server.
  • Size broadcast → spikeOnServerSize → plane re-presented smaller → container smaller → observer fires → … one step down per server round-trip, mirrored by every client on the shared PTY.

Why it was new: the loop's return edge used to be dead — the Round-4 TypeError aborted applyServerSize before spikeOnServerSize could re-present the plane, so the cycle broke after one pass. Fixing the crash completed the circuit.

Fix: while a pane rides a 3D plane (isRidingSpikePlane), the 2D fit machinery stands down — the ResizeObserver refit and the socket-open fit are skipped (fitting a grid-derived box back onto the grid is circular by construction). Soft votes still flow (the current grid is the truth in 3D); world-close restore is unaffected (spikeOpen is already false there).

Outcome: in the served dev bundle; verify by relaunching the spike app — panes should hold size on world entry, and the 2D clients should stay still.
ROUND 9

Session 2: real reformat, fluid resize, the last ratchets, and the box that lied by 22px

verified in-app
  • `r` reformat, three designs later. The R3 snap-to-standard-box yanked panes to a wildly different size — user decree: a 3D reformat must never change the pane's physical size. Replaced with re-pin-current-grid — which turned out to be end-to-end invisible: the kernel raises SIGWINCH only on a *real* size change, and the server's size StateFlow dedups equal values, so a same-size ForceResize reaches nobody (why "reformat never worked in 3D", ever). Final design: the jiggle (tmux's redraw trick) — force the PTY one row off and back (REFORMAT_JIGGLE_MS), two real SIGWINCHes, full program repaint at the current grid, then pin the viewport to the tail. The clue that unmasked it: a grid-key width change "auto-reformats" — because it is a real change.
  • Grid keys: bigger steps, fluid glide. ±20 cols / ±10 rows per press, pane box eases over GRID_ANIM_MS (260 ms CSS width/height transition; terminal reflows instantly, the box clips/reveals; border SVG stretches along). Detour: ⌃-arrows are swallowed system-wide by macOS Mission Control; ⌥-arrows never reached the world handler in the real app — reverted to , . < >.
  • Two more shrink-ratchet paths gated. Every Mac-window resize fired the toolkit's onGeometryChanged → forceReassert on all visible terminals — fitting spike-riding panes to their grid-derived boxes, one shrink step per resize event. The gate now lives inside forceReassert (covers fonts-loaded, tab-activation edge, ⌃⌥R too); world-close restore unaffected (spikeOpen clears first).
  • The pane box must budget ALL the chrome. presentPaneToGrid sized the box to bare grid × cell; the real DOM adds the xterm element's padding + scrollbar gutter and .terminal-inner's 10/4/12/12 breathing-room strip (stylesheet padding survives inline cssText; it lives on an intermediate element, so a container-only read missed it — walk the whole chain and sum every level). Without it the screen sat 10px low and the bottom line clipped ("the peeking cursor"), immune to `r` by construction. Diagnosed by the user pasting a measuring snippet into the dev-window DevTools — screenBottomOverflow was the single truth-telling number.
  • Open-time settle passes. The world-open present runs before the CSS3D subtree is styled (computed padding reads 0 → box 22px short) and a mirror's ring replay outruns the first tail pin ("no cursor on launch"). Two settle passes (120/700 ms) re-present + re-pin.
  • OOB overlay gated off on planes. Its getBoundingClientRect math reads through the CSS3D transform — after `r` it painted a bogus hatch strip over live content. Hidden + skipped while riding ([isRidingSpikePlane]).
Outcome: user-confirmed: grid keys resize fluidly at the new steps, `r` visibly reformats without changing pane size, window resizes no longer shrink panes, and panes open at the correct box with the cursor settling into view. Process lesson (now in memory): during long operations, report progress continually.

  • Panes not mounted in this window's 2D layout are mirrors — a second live client on the same PTY socket, fed a one-time ring replay. Engaging promotes them to the real terminal.
  • A diagnostic WebSocket probe (pty-probe.mjs, in the session scratchpad) can attach as a second client and dump post-ForceResize bytes; it was blocked on extracting the auth token, but can be run manually.
  • Another Claude session is concurrently editing this same working tree (it split World3DSpike.kt into modules mid-session) — and its leftover headless Chrome was the Round-7 phantom client. Lesson: before diagnosing any multi-client resize weirdness, run lsof -nP -iTCP:8443 -sTCP:ESTABLISHED and account for every connection.
