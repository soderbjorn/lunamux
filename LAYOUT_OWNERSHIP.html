Who owns layout in Termtastic?

Server WindowConfig vs. the toolkit's LAYOUT_STATE blob — the two parallel layout representations, which one is authoritative on web, and the trap that caused the 3D pane-order bug.
reference note discovered: 2026-07-06 branch: world3d-spike bug fixed

1. TL;DR

Layout has two representations. The server's typed WindowConfig is authoritative for mobile/native. On web, the darkness-toolkit's LAYOUT_STATE blob is what actually renders — pane order, geometry, minimize and preset all flow through the toolkit, not through WindowConfig. The server still persists and rebroadcasts that blob, but only as an opaque flat-KV entry inside /api/ui-settings; it does not model it. Two fields exist only in the toolkit blob with no typed server mirror at all: pane order / "primary" and isMinimized.

2. The two stores

Concern Server WindowConfig (typed) Toolkit LAYOUT_STATE (opaque blob) Authoritative for web?
Tab structure, pane existence, sessions ✅ modelled — Server
Pane geometry (x / y / w / h / z, maximized) ✅ fields on Pane ✅ geometryByTab Toolkit (server copy driven only by mobile)
Pane order within a tab / "primary" ❌ none — panes is creation-ordered, plus one focusedPaneId ✅ paneOrderByTab (head = primary) Toolkit only
isMinimized ❌ no field ✅ geometryByTab[..].isMinimized Toolkit only
Layout preset per tab ✅ TabConfig.layoutPreset ✅ presetByTab Toolkit (server copy is mobile/demo mirror)
Active tab, focused pane, hidden flags ✅ modelled — Server

Beyond layout, the same flat-KV persister also carries genuine client UI prefs — theme selection, collapsed sidebar sections, sidebar font/size, hotkeys. Those are expected to be client-side and are not part of this story.

3. What "the server owns layout" actually means

The toolkit persists its blob through the host Persister. In termtastic that adapter round-trips to the server's flat-KV: the blob is merged into termtastic.json and rebroadcast over the /window socket (cross-client sync, issue #58). So the server gives the blob durability and rebroadcast — but it never parses it into WindowConfig. "Server-owned" = that KV round-trip, not a structured field the server reasons about.

web toolkit  →  persistLayoutState()  →  Persister  →  /api/ui-settings (flat-KV)  →  server merge + broadcast  →  rawLayoutState  →  every client

4. Proof: who sends the structured commands

The server does have structured layout commands — SetPaneGeom, MovePaneWithinTab, ToggleMaximized, ApplyLayout. Grepping their senders is the tell:

  • SetPaneGeom / MovePaneWithinTab are sent by the native/mobile LayoutViewModel (client/commonMain) and the demo server — never by any web code.
  • On web, geometry / order / minimize / re-tile all go through the toolkit's LayoutController + persistLayoutState().

So the WindowConfig.Pane geometry fields and those commands are vestigial for web, live for mobile. (Mobile has no toolkit, so it reads the same broadcast LAYOUT_STATE blob by parsing it directly in WindowStateRepository — see the minimize-to-mobile pattern.)

5. The bug this caused (world3d-spike)

Symptom
User manually rearranged panes in a tab. The left sidebar showed the new order; the 3D "world" ring showed the original order — with a terminal wedged between the git and folder panes.
Cause
The ring built its order from the server's tab.panes (creation order). The sidebar renders panes sorted by the toolkit's paneOrderByTab (the user's drag-reorder). A sidebar drag mutates only ctl.paneOrder + persists — it never reorders WindowConfig.panes. So the two diverged, and the ring read the wrong store.
Fix
World3DSpike.collectPaneSpecs now sorts each tab's panes by a new toolkitPaneOrder(tabId) helper, which reads LAYOUT_STATE.paneOrderByTab off termtasticClient.windowState.rawLayoutState — the same source and the same stable-sort-with-untracked-at-tail rule the sidebar uses (AppShellMount.buildLeftSidebarContent). The ring now matches the sidebar.

6. The rule of thumb

On web, anything that wants the user's pane order must read paneOrderByTab (via rawLayoutState), not tab.panes. Reading tab.panes gives you stale creation order. The same goes for pane minimize state (geometryByTab[..].isMinimized) and, for what actually renders, geometry and preset.

How to read it (Kotlin/JS)

// LAYOUT_STATE.paneOrderByTab : { tabId -> [paneId, paneId, ...] }  (head = primary)
private fun toolkitPaneOrder(tabId: String): List<String> {
    val raw = termtasticClient.windowState.rawLayoutState.value ?: return emptyList()
    val obj = when {
        raw is JsonObject -> raw
        raw is JsonPrimitive && raw.isString -> Json.parseToJsonElement(raw.content).jsonObject
        else -> return emptyList()
    }
    val orderByTab = obj["paneOrderByTab"] as? JsonObject ?: return emptyList()
    val arr = orderByTab[tabId] as? JsonArray ?: return emptyList()
    return arr.mapNotNull { it.jsonPrimitive.content }
}
// Then, mirroring the sidebar (stable sort, untracked panes to the tail):
val rank = toolkitPaneOrder(tabId).withIndex().associate { (i, id) -> id to i }
val ordered = tab.panes.sortedBy { rank[it.leaf.id] ?: Int.MAX_VALUE }

The blob arrives either as a JsonObject or as a JSON-string JsonPrimitive depending on the persistence round-trip — handle both.

7. Open decision (spike ⇧-reorder)

The spike's in-ring reorder (⇧←/⇧→ → movePaneSlot) still sends MovePaneWithinTab, i.e. it writes the server tab.panes — the store web ignores. After the read-fix above, that gesture is now visually inert on web. To make it work + stay consistent it should write the toolkit order (read-modify-write LAYOUT_STATE.paneOrderByTab via webSettingsPersister), exactly like the sidebar's own drag-reorder does. Left unwired pending a call on whether pane order should eventually become server-authoritative instead.
